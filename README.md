# Parallel Life Simulator

> What could your money have built by now?

A personal finance tool that ingests your real bank statement, categorizes every transaction using a two-stage keyword + machine learning pipeline, benchmarks your spending against real BLS Consumer Expenditure data, ranks your top savings opportunities, and projects compound wealth growth across three parallel scenarios using Monte Carlo simulation.

**Live demo:** https://parallel-life-hiya.hub.zerve.cloud
**Built with:** Python · Streamlit · pandas · numpy · matplotlib · scikit-learn · Zerve AI

---

## What it does

1. **Ingests** CSV exports from Chase, Amex, Bank of America, Mint, and YNAB — or OFX/QFX files
2. **Categorizes** merchants using a two-stage pipeline: keyword matching first, a trained TF-IDF + Logistic Regression model second for anything the keyword stage can't resolve
3. **Excludes** rent, payroll deposits, transfers, and loan payments from spend analysis entirely
4. **Benchmarks** your spending against real BLS Consumer Expenditure Survey 2023 national averages (honestly scoped — see below)
5. **Ranks** discretionary categories by savings potential using `monthly_spend × reducibility_weight`
6. **Simulates** 1,000+ Monte Carlo wealth paths per scenario using log-normal returns
7. **Exports** results as a downloadable PDF report or CSV data file

## Categorization: keyword matching + ML fallback

### How it works

1. `classifier.infer_category()` runs first — fast, deterministic, interpretable keyword matching against 200+ merchant terms across 8 categories
2. If that returns `"Other"`, `ml_classifier.categorize_with_fallback()` runs second — a trained TF-IDF (character n-grams) + Logistic Regression model
3. The ML model only overrides `"Other"` if its confidence exceeds a threshold (0.12, selected via a held-out validation split — see "Threshold selection" below); otherwise it stays `"Other"` rather than forcing a low-confidence guess

### Measured accuracy — methodology

This is the part most personal-finance side projects skip. The full process:

1. **250 manually labeled transactions** (`validation.csv`) — including deliberately hard cases: payment-processor prefixes (`SQ*`, `TST*`, `PAYPAL*`), abbreviated merchant names (`WM SUPERCENTER`, `AMZN MKTP`), hyphenated variants (`WAL-MART`)
2. **Split 70/30** into `train_labels.csv` (175 rows) and `test_labels.csv` (75 rows) — the test set is never touched during training
3. **Threshold selection on a separate validation split** of the training data (140/35), NOT on the test set — this avoids the common mistake of tuning a hyperparameter against the same data used to report the final accuracy number
4. **Final evaluation runs once** on the untouched test set

Run it yourself:
```bash
python3 ml_classifier.py          # trains the model on train_labels.csv
python3 evaluate_classifier.py    # evaluates on test_labels.csv, prints confusion matrix
```

### Results (measured, not asserted)

| Pipeline | Accuracy on 75 held-out transactions |
|---|---|
| Keyword-only | 76.0% |
| Keyword + ML fallback | **85.3%** |

This is a real **+9.3 percentage point** improvement from adding the ML stage, measured honestly. The full confusion matrix and per-category precision/recall/F1 are in `evaluate_classifier.py`'s output — macro-average F1 improves from 0.78 to 0.82.

**Known limitation, stated honestly:** with only 175 training examples, the model's confidence scores run low even when its prediction is correct (e.g. `APPLE.COM/BILL` → Shopping, correct, confidence 0.178). The threshold of 0.12 was chosen because validation accuracy is flat across 0.05–0.12 before dropping off — this is a small-data classifier, not a production-scale one, and the README says so rather than implying otherwise.

## Fixed/Excluded category

Rent, mortgage payments, payroll deposits, and account transfers are detected and removed from spend analysis entirely — they are never treated as a discretionary expense and never appear as a "savings opportunity." Without this, the tool would give wrong advice (e.g. "cut your rent by 30%") the moment a real bank statement is uploaded instead of demo data. `rank_scenarios()` hard-skips any category with reducibility 0 — covered by explicit unit tests in `test_engine.py`.

## BLS national benchmark — what it does and doesn't claim

The app compares your spending against real BLS Consumer Expenditure Survey 2023 figures: national average annual expenditure ($77,280/household) and category shares from the published "Consumer Expenditures in 2023" report.

**What this honestly does NOT claim:** an earlier version of this README described comparing spending against "BLS median for your income bracket." That claim was removed because BLS's full category-by-quintile dollar breakdown (their Table 2) exists only as a published table on bls.gov, not as structured data accessible during development. Building income-quintile-specific dollar comparisons without that table would mean fabricating numbers that look like real BLS data but aren't. Instead, `benchmark.py`:

- Uses only the **national average** category shares (real, sourced, cited)
- Classifies the user's income quintile correctly using real 2023 BLS bounds, for context — but does NOT use quintile-specific category dollar amounts, because that data wasn't available
- Only benchmarks categories with a clean 1:1 mapping to a BLS major category (Food & Dining, Transportation, Health & Wellness, Entertainment) — Shopping, Subscriptions, and Utilities are intentionally NOT benchmarked, because BLS splits those across multiple major categories and a forced mapping would be misleading

This is a smaller, defensible claim ("you spend more than the average household on dining out") instead of a larger, unverifiable one.

## Monte Carlo simulation

- **Model:** log-normal monthly returns with Itô correction
- **Volatility:** 15.6% annual (S&P 500 historical, Damodaran 2024)
- **Mean return:** user-adjustable 4–12%, default 7% (S&P 500 30-year nominal average, Shiller dataset)
- **Output:** 10th/25th/median/75th/90th percentile bands per scenario, probability of doubling total contributed capital
- `run_monte_carlo()` is cached via `@st.cache_data` keyed on `(monthly_saving, years, rate, n_sims)`

---

## Persistence

- **PDF/CSV export** (`report_export.py`): a 3-page PDF (summary + scenarios + top merchants) and a CSV data file, both downloadable via `st.download_button` — the analysis survives closing the tab
- **Session state caching**: parsing and categorization (including the ML model call) only re-run when the uploaded file actually changes, not on every slider drag — adjusting the return-rate slider doesn't trigger a full re-parse

## Testing & CI

```bash
python3 test_engine.py
```

51 unit tests covering: compound growth boundary cases, classifier edge cases, `rank_scenarios` with single/zero/all-Fixed-category inputs, CSV/OFX parsing failure modes, Monte Carlo regression tests (the three bugs fixed in an earlier version: a stubbed `prob_double`, an unwired `n_simulations` parameter, a dead `monte_carlo_stats()` call), ML fallback classifier tests, and benchmark module tests confirming it never fabricates a number for unmapped categories.

`.github/workflows/test.yml` runs the full suite (training the ML model, then `test_engine.py`, then `evaluate_classifier.py`) on every push to `main`, across Python 3.10 and 3.11.
